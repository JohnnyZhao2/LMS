项目结构规则总结
核心目录结构
采用特性驱动的架构，主要代码位于 src 文件夹中

src/  
├── app/          # 应用层：路由、主组件、提供者、路由器  
├── assets/       # 静态资源：图片、字体等  
├── components/   # 全局共享组件  
├── config/       # 全局配置、环境变量  
├── features/     # 特性模块（主要代码）  
├── hooks/        # 全局共享 hooks  
├── lib/          # 预配置的可重用库  
├── stores/       # 全局状态存储  
├── testing/      # 测试工具和模拟数据  
├── types/        # 共享 TypeScript 类型  
└── utils/        # 共享工具函数  
特性模块组织
每个特性模块遵循内部结构化组织

src/features/awesome-feature/  
├── api/          # 特定 API 请求和 hooks  
├── assets/       # 特定静态资源  
├── components/   # 特定组件  
├── hooks/        # 特定 hooks  
├── stores/       # 特定状态存储  
├── types/        # 特定类型定义  
└── utils/        # 特定工具函数  
注意：并非每个特性都需要所有文件夹，只包含必要的文件夹 

架构边界规则
1. 特性隔离
禁止跨特性导入，保持特性独立性
在应用层组合不同特性，而非直接导入
2. 单向代码流
代码流向：共享模块 → 特性 → 应用：

共享模块可被特性和应用导入
特性只能从共享模块导入
应用可从特性和共享模块导入
特性不能导入其他特性
共享模块不能导入特性或应用代码

最佳实践
代码就近原则：将组件、函数、样式、状态等尽可能靠近使用位置 

避免大型组件：不要在组件内嵌套多个渲染函数，应提取为独立组件

文件命名规范：使用 kebab-case 命名文件和文件夹 

绝对导入：配置并使用绝对导入，避免复杂的相对路径 

避免 Barrel 文件：直接导入文件而非使用 barrel 文件，以支持 Vite 的 tree shaking 

项目标准
项目强制执行以下标准 

ESLint：代码质量和编码标准
Prettier：代码格式一致性
TypeScript：类型安全
Husky：Git hooks，提交前验证
Notes
这些结构规则旨在创建可扩展、可维护且易于导航的代码库。该架构框架无关，可应用于 React、Next.js、Remix 或 React Native 应用。实际实现中，不同框架（如 Next.js Pages Router）可能需要适当调整，但核心原则保持不变

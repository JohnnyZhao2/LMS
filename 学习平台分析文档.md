# LMS 学习管理系统 - 开发需求文档

## 1. 项目概述

本文档旨在为 LMS（Learning Management System）学习管理系统的重新开发提供一份全面的需求和技术规范。该系统是一个基于 Django REST Framework 和 React 的全栈应用，旨在提供一个集“学、练、考、评”于一体的企业级在线学习平台。通过对现有代码库的梳理，本文档将详细阐述系统的核心功能模块、数据结构、API 接口和业务逻辑，并为后续的重构和开发工作提供清晰的指引。

**核心目标**：

- **模块化与解耦**：构建清晰、低耦合的模块化架构，便于独立开发、测试和维护。
- **统一与规范**：建立统一的数据模型、API 接口标准和编码规范，提升开发效率和代码质量。
- **角色与权限**：实现灵活且强大的角色权限控制体系，满足不同用户（学员、导师、管理员等）的操作需求。
- **版本化管理**：对核心教学资源（知识、题目、试卷）进行版本控制，确保教学内容的可追溯性和迭代能力。

## 3. 核心概念与设计

### 3.1. 统一的用户与角色模型

系统以一个统一的 `User` 模型为中心，通过多对多的 `Role` 模型实现灵活的权限控制。每个用户可以拥有一个或多个角色。角色之间存在一个预定义的优先级顺序 (`ROLE_PRIORITY_ORDER`)，用于在用户拥有多角色时确定其默认的最高权限。

- **核心角色**：
  - `STUDENT` (学员): 系统的主要使用者，参与学习任务。
  - `MENTOR` (导师): 负责指导学员，创建任务，批改作业。
  - `DEPT_MANAGER` (室经理): 部门管理者，拥有对本部门学员和导师的管理权限。
  - `TEAM_MANAGER` (团队经理): 跨部门管理者，拥有更广泛的数据查看和分析权限。
  - `ADMIN` (管理员): 拥有系统的最高管理权限。

- **权限继承与数据隔离**：权限系统遵循层级关系，高层级角色通常包含低层级角色的权限。数据访问权限根据角色进行严格隔离，例如，导师只能查看其名下学员的数据，室经理只能查看本部门的数据。

### 3.2. 版本化的资源管理

为了确保教学内容的严谨性和可追溯性，系统的核心资源（知识、题目、试卷）采用版本化设计。对这些资源的任何修改都会创建一个新的版本，而不是直接在原版本上修改。

- **工作机制**：
  - 每个资源实例由 `resource_uuid` (资源唯一标识) 和 `version_number` (版本号) 共同确定。
  - `is_current` 字段标记了某个 `resource_uuid` 的最新（当前正在使用）的版本。
  - 当一个资源被编辑时，系统会克隆当前版本，生成一个版本号+1的新记录，并将旧版本的 `is_current` 设为 `False`。
  - 任务等业务实体直接关联到资源的特定版本，确保即使资源更新，已发布的任务内容也不会改变。

### 3.3. 统一的 API 响应格式

所有 API 接口遵循统一的响应结构，以简化前端处理逻辑并提供一致的错误处理机制。

- **成功响应**: `{"code": "SUCCESS", "message": "success", "data": { ... }}`
- **错误响应**: `{"code": "ERROR_CODE", "message": "Error description", "details": { ... }}`
- **分页响应**: 成功响应的 `data` 字段内包含 `count`, `total_pages`, `current_page`, `results` 等分页信息。


### 3.4. 后端分层架构

后端采用清晰的分层架构，将业务逻辑、数据访问和接口展示分离，确保代码的可维护性和可测试性。

| 层级 | 职责 | 对应文件 | 说明 |
| --- | --- | --- | --- |
| **View 层** | 接收 HTTP 请求，参数校验，调用 Service，返回响应 | `views.py` | 继承 `BaseAPIView`，通过 `self.service` 自动注入 Service 实例 |
| **Service 层** | 编排业务逻辑，处理权限校验和事务 | `services.py` | 继承 `BaseService`，通过构造器注入 `request`，内部通过 `self.user` 访问当前用户 |
| **Selector 层** | 复杂查询逻辑的封装 | `selectors.py` | 纯函数，负责数据库查询的拼装和优化 |
| **Model 层** | 数据模型定义 | `models.py` | 继承公共 Mixin（`TimestampMixin`, `SoftDeleteMixin`, `VersionedResourceMixin` 等） |
| **Serializer 层** | 数据序列化与反序列化 | `serializers.py` | 负责请求体校验和响应体格式化 |

### 3.5. 公共基础设施 (Core)

`core` 模块提供了所有业务模块共享的基础设施，包括：

- **基类 (Base Classes)**：`BaseService`（提供 `validate_not_none`, `validate_permission`, `check_published_resource_access` 等通用方法）和 `BaseAPIView`（提供 Service 自动注入）。
- **数据模型 Mixin**：
  - `TimestampMixin`：自动维护 `created_at` 和 `updated_at` 字段。
  - `SoftDeleteMixin`：提供软删除功能（`is_deleted`, `deleted_at`）和 `restore()` 恢复方法。
  - `CreatorMixin`：跟踪资源创建者，并设置 `on_delete=models.PROTECT` 防止创建者被误删。
  - `VersionedResourceMixin`：提供版本化管理功能，并包含 `next_version_number()` 和 `clone_new_version()` 等辅助方法。
- **数据权限 Mixin**：
  - `DataScopeMixin`：根据用户角色自动过滤查询集（管理员全部、室经理本部门、导师本人学员）。
  - `StudentDataScopeMixin`：专用于 User 模型的数据权限过滤。
  - `TaskDataScopeMixin`：专用于任务模型的数据权限过滤（额外包含创建者视角）。
- **统一响应**：`success_response`, `created_response`, `no_content_response`, `error_response`, `paginated_response`, `list_response`。
- **异常处理**：`BusinessError` 业务异常类和 `ErrorCodes` 错误码常量，通过全局的 `custom_exception_handler` 和视图层的 `BusinessErrorHandlerMixin` 统一处理。系统内部维护了业务错误码到 HTTP 状态码的详细映射关系。
- **分页器**：`StandardResultsSetPagination`（默认 20 条/页，最大 100）和 `SmallResultsSetPagination`（默认 10 条/页，最大 50）。
- **限流器**：`AuthThrottle`（认证接口 10次/分钟）、`SubmissionThrottle`（提交接口 30次/小时）、`BurstRateThrottle`（60次/分钟）。
- **日志装饰器**：`@log_user_action`, `@log_content_action`, `@log_operation`，用于在业务方法上自动记录操作日志。装饰器接受 `action_key` 参数，与 `ActivityLogPolicy` 联动，实现日志记录的可配置化。
- **工具函数**：`generate_random_password`, `get_current_time`, `is_past_deadline`, `is_within_time_window`。
- **查询参数解析**：`query_params.py` 提供 `parse_int_query_param` 和 `parse_bool_query_param`，统一处理查询参数的类型转换和范围校验，校验失败时抛出 `BusinessError`。

### 3.6. 业务规则汇总

以下是系统中已确认的核心业务规则，在重新开发时必须严格遵守：

1. **新用户默认角色**：新创建的用户自动分配 `STUDENT` 角色，且该角色不可被移除。
2. **角色互斥约束**：`ADMIN`, `DEPT_MANAGER`, `TEAM_MANAGER` 为管理角色，拥有管理角色时不再保留 `STUDENT` 角色。同时，`DEPT_MANAGER` 和 `TEAM_MANAGER` 角色本身也互相斥。
3. **角色唯一性**：每个部门只能有一个室经理，全局只能有一个团队经理。
4. **用户删除限制**：仅允许删除已停用（离职）用户。删除时会物理删除该用户创建的所有关联数据（任务、知识、试卷、题目、提交记录等），是高风险操作。
5. **资源版本化**：知识、题目、试卷的编辑操作创建新版本，不修改原版本。
6. **资源访问控制**：非管理员只能访问 `is_current=True` 的资源版本。
7. **试卷提交规则**：练习试卷可多次提交，考试试卷只能提交一次。
8. **自动评分**：客观题（单选、多选、判断）提交后自动评分，主观题（简答）需要导师手动评分。
9. **数据权限隔离**：导师只能查看其名下学员的数据，室经理只能查看本部门的数据，管理员和团队经理可查看所有数据。
10. **软删除机制**：知识、题目、试卷、任务等核心资源采用软删除，不物理删除。

#### 4.1.3. 补充业务规则

- **角色分配约束**: `DEPT_MANAGER` (室经理) 和 `TEAM_MANAGER` (团队经理) 角色互斥；超级管理员 (`is_superuser`) 只能拥有 `ADMIN` 角色。
- **导师分配校验**: 分配导师时，需校验对方是否拥有 `MENTOR` 角色、账户是否激活，且禁止将用户自己设为自己的导师。
- **用户停用限制**: 禁止停用超级管理员账户。
- **角色切换**: 系统支持通过请求头 `X-Current-Role` 来让多角色用户在不同身份间切换，获取对应角色的权限和数据视图，无需重新登录。

#### 4.1.5. 技术实现细节

- **自定义用户模型管理器 (`UserManager`)**: `User` 模型使用自定义的 `UserManager`，支持使用 `employee_id` 作为唯一标识和登录字段。
- **角色感知的 JWT 认证**: 自定义的 `RoleAwareJWTAuthentication` 认证后端，会解析 JWT 中的 `current_role` 声明，并将其附加到 `request.user` 对象上，是实现角色切换功能的基础。
- **数据权限实现**: `permissions.py` 中的 `get_accessible_students` 和 `filter_queryset_by_data_scope` 等函数是数据权限隔离的核心，它们根据当前用户的角色和数据范围构建动态查询条件。
- **管理命令 (`init_data`)**: 提供 `python manage.py init_data` 命令，用于一键初始化系统的基础数据，包括默认部门、所有角色以及一个默认的管理员账号。
- **JWT 令牌结构**: 登录成功后返回的 JWT 令牌中，包含了 `employee_id`, `username`, `roles` (所有角色), `current_role` (当前生效角色) 等自定义声明。

#### 4.2.3. 补充业务规则

- **差异化更新策略**: 更新历史版本 (`is_current=False`) 时，会直接在原版本上修改（用于修正错误）；而更新当前版本时，则会创建一个新版本。

- **逾期自动标记**: 对于未完成的任务，系统会通过定时任务或在访问时检查，将其状态自动标记为 `OVERDUE`。
- **异常学习行为检测**: 系统会根据预设规则（如学习时长过短）识别并标记异常学习行为，供导师参考。

#### 4.5.3. 补充业务规则

- **考试计时与恢复**: 系统通过 `remaining_seconds` 字段实现考试的倒计时功能，并支持在用户中断答题（如关闭浏览器）后，重新进入时恢复计时。
- **断点续作**: 系统支持用户重新进入未提交的答题会话，之前保存的答案会被加载。
- **成绩更新逻辑**: 对于允许多次尝试的练习，系统会在每次提交后，更新 `TaskAssignment` 中的 `score` 为历史最高分。
- **任务完成状态联动**: 学员提交试卷后，系统会检查该任务下的所有要求（知识学习、其他试卷）是否均已完成，如果全部完成，则自动更新整个任务的状态为 `COMPLETED`。
- **答案归属验证**: 后端在保存答案时，会严格校验提交的答案是否属于当前用户正在进行的 `Submission`，防止非法提交。

#### 4.5.5. 评分模块补充业务规则

- **最新提交定义**: 在获取学员答案进行评分时，系统通过 `attempt_number` 降序、`submitted_at` 降序来确定唯一的最新的提交记录进行展示。
- **题目通过率计算**: 客观题通过率为“正确数 / 总提交数”；主观题通过率为“得分率>=60%的已评分答案数 / 总已评分答案数”，未评分的答案不计入统计。
- **评分分数校验**: 提交的评分分数必须在 0 分到该题目的满分之间。
- **批量评分事务处理**: 批量评分的整个过程包裹在数据库事务中，保证数据一致性。
- **判断题选项构造**: 对于没有预设选项的判断题，后端会自动生成“正确”和“错误”两个选项用于前端展示和数据分析。

### 4.7. 仪表盘 (Dashboard)

#### 4.7.1. 模块概述

仪表盘是系统的门户，根据不同角色的需求，展示最核心的统计数据和快捷入口。它是数据可视化的集中体现，旨在帮助用户快速了解当前的学习或管理状态。该模块本身不拥有独立的数据模型，而是通过聚合查询其他模块（任务、提交、抽查等）的数据来生成统计结果。

#### 4.7.2. 补充业务规则与指标定义

- **紧急任务**: 默认将 **48 小时**内截止的任务定义为紧急任务。
- **同伴排名**: 学员的同伴排名是基于**同一导师下**的所有学员的任务完成率进行排序。
- **风险学员**: 完成率低于 **60%** 或平均分低于 **60分** 的学员被定义为“风险学员”。
- **需关注学员**: 综合多个维度（任务完成率 < 70%, 平均分 < 60, 30天未活跃）来识别需要关注的学员。
- **月度趋势**: 系统会统计最近 **6 个月**的月度平均分和提交数趋势。

### 4.8. 活动日志 (Activity Logs)

#### 4.8.1. 模块概述

## 5. 前端模块设计 (Frontend)

前端采用基于 React 和 TypeScript 的现代化技术栈，通过 Vite 进行构建。UI 层面，采用 `shadcn/ui` 组件库，它提供了基于 Tailwind CSS 的高度可定制化的基础组件，能够快速构建出美观且一致的用户界面。

### 5.1. 整体架构

- **组件化**: 遵循原子设计理念，构建可复用的 UI 组件，从原子 (Button, Input) 到分子 (Form, Card) 再到组织 (Header, Sidebar)，最终构成完整的页面模板。
- **状态管理**: 
  - **服务端状态**: 使用 `React Query` (`TanStack Query`) 统一管理所有与后端 API 的交互，包括数据获取、缓存、同步和错误处理。
  - **客户端状态**: 对于全局性、跨组件共享的客户端状态（如当前用户信息、主题切换等），建议使用 `Zustand` 或 `React Context` 进行管理。
- **路由管理**: 使用 `React Router` 实现单页应用 (SPA) 的路由功能，并根据用户角色进行路由守卫，控制页面访问权限。
- **代码规范**: 采用 ESLint 和 Prettier 保证代码风格和质量的一致性。

### 5.2. 页面模块与路由设计

前端页面和路由的设计严格遵循角色驱动的原则。系统通过 URL 路径中的 `/:role` 参数来区分不同角色的工作区，并动态渲染对应的侧边栏菜单和仪表盘。所有核心业务功能都通过 `ProtectedRoute` 组件进行权限保护。

#### 5.2.1. 核心路由结构

-   **认证路由**: `/login` - 独立的登录页面。
-   **角色工作区**: `/:role/*` - 所有业务页面的根路径，例如 `/student/dashboard`, `/admin/users`。
-   **路由守卫**:
    -   `RoleRouteWrapper`: 校验 URL 中的角色是否有效，以及当前登录用户是否拥有该角色权限。
    -   `ProtectedRoute`: 进一步对具体页面进行权限控制，确保只有特定角色的用户可以访问。

#### 5.2.3. 各角色侧边栏菜单配置

不同角色登录后，侧边栏展示的菜单项不同，具体配置如下：

| 角色 | 侧边栏菜单项 |
| :--- | :--- |
| **学员 (`STUDENT`)** | 概览 → 知识中心 → 任务中心 → 个人中心 |
| **导师 (`MENTOR`)** | 概览 → 试卷中心 → 任务中心 → 抽查中心 |
| **室经理 (`DEPT_MANAGER`)** | 概览 → 试卷中心 → 任务中心 → 抽查中心 |
| **团队经理 (`TEAM_MANAGER`)** | 概览 → 数据看板 |
| **管理员 (`ADMIN`)** | 概览 → 知识管理 → 试卷管理 → 任务管理 → 用户管理 |

> **说明**：评分中心和日志策略管理等页面虽然有对应的路由，但不在侧边栏菜单中直接展示，而是通过仪表盘中的快捷入口或其他页面内的链接进入。
